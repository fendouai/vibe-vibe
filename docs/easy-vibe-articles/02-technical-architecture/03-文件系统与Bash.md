# 用文件系统和 Bash 构建智能体

> **作者**：Ashka Stephen (Vercel)  
> **来源**：https://vercel.com/blog/how-to-build-agents-with-filesystems-and-bash  
> **核心观点**：简单接口比专用框架更适合智能体

---

## 成本对比

| 方案            | 单次通话成本 |
| --------------- | ------------ |
| 复杂工具链      | ~$1.00       |
| 文件系统 + Bash | ~$0.25       |
| **节省**        | **75%**      |

> 输出质量还更好了

---

## 为什么文件系统适合上下文管理

| 优势                 | 说明                                |
| -------------------- | ----------------------------------- |
| **结构天然贴合领域** | 层级关系直接映射目录结构            |
| **检索结果精确**     | grep 返回明确匹配，不是"看起来相关" |
| **上下文最小化**     | 按需加载，不一开始塞进提示          |

---

## 智能体工作流程

```
智能体收到任务
    ↓
探索文件系统（ls、find）
    ↓
搜索相关内容（grep、cat）
    ↓
发送上下文 + 请求给 LLM
    ↓
返回结构化输出
```

## 为什么优先考虑 Bash 和文件系统

| 原因                 | 说明                             |
| -------------------- | -------------------------------- |
| **贴近模型原生熟悉** | grep/cat/find/awk 已在训练数据中 |
| **面向未来架构**     | 模型编码能力提升，系统同步受益   |
| **更容易调试**       | 直接看到读了什么、执行了什么     |
| **隔离获得安全**     | 沙箱开放推理空间，不开放无限权限 |
| **减少维护代码**     | 不必为每种数据写检索管道         |

---

## 关键结论

> **最好的架构未必是"功能最多"的架构，而是几乎没有多余抽象、只保留基础接口的架构**
