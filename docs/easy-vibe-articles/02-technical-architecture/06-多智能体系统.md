# 多智能体系统如何支撑 AI 原生工程

> **作者**：Spiros Xanthos, Gabor Angeli, Bharat Khandelwal (Resolve AI)  
> **来源**：https://resolve.ai/blog/multi-agent-systems-ai-native-engineering  
> **核心观点**：专业分工、协作边界与系统组织方式

---

## AI 辅助 vs AI 原生

| 模式            | 特点                                              |
| --------------- | ------------------------------------------------- |
| **AI 辅助工程** | AI 提高效率，工作流仍以人为中心                   |
| **AI 原生工程** | AI 是处理工作的主入口，提出目标→系统组织→返回结果 |

---

## 生产系统的复杂性

| 调查维度       | 说明           |
| -------------- | -------------- |
| 微服务追踪链路 | 关联几十个服务 |
| 慢查询分析     | 连接池耗尽     |
| 部署变更检查   | 基础设施变更   |
| 安全事件排查   | 认证日志异常   |
| 扩缩容评估     | 负载模式       |
| SLA 影响       | 客户支持工单   |

> 需要分工明确、上下文各自充分的专业分析

---

## 方法演进与限制

| 方法           | 失效原因                         |
| -------------- | -------------------------------- |
| **LLM**        | 单次生成，无反馈闭环，无系统集成 |
| **LLM + 工具** | 上下文窗口有限，缺少持久状态     |
| **单一智能体** | 调查只能顺序推进，易卡在错误假设 |
| **多智能体**   | 需要额外协调成本                 |

---

## 多智能体系统的价值

| 价值         | 说明                   |
| ------------ | ---------------------- |
| **统一协调** | 主智能体编排整体流程   |
| **领域分工** | 每个智能体专注特定领域 |
| **并行推进** | 多个分析线程同时进行   |
| **假设验证** | 并行验证假设，快速收敛 |

---

## 构建多智能体系统的难点

| 能力         | 说明                                       |
| ------------ | ------------------------------------------ |
| **领域理解** | 不了解生产系统真实问题，无法设计智能体边界 |
| **AI 工程**  | 上下文传播、并行协调、冲突解决、失败重试   |
| **交叉突破** | 领域知识 + AI 架构能力结合                 |

---

## Resolve AI 定位

| 角色              | 说明                         |
| ----------------- | ---------------------------- |
| **全天候 AI SRE** | 事件响应与生产系统运维       |
| **客户**          | DataStax、Tubi、Rappi        |
| **价值**          | 让工程师把精力放回到构建本身 |

---

## 关键结论

> **多智能体不是把单智能体复制几份，而是把专业分工、协作边界与系统组织方式重新带回软件工程**
