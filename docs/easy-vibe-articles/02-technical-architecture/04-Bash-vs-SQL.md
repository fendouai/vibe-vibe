# Bash vs SQL 智能体对比实验

> **作者**：Ankur Goyal, Andrew Qu (Vercel/Braintrust)  
> **来源**：https://vercel.com/blog/testing-if-bash-is-all-you-need  
> **核心观点**：工具选择需要被系统性验证

---

## 实验设置

| 数据   | GitHub issues + pull requests |
| ------ | ----------------------------- |
| 智能体 | Claude Sonnet 4.5             |
| 问题   | 从简单到复杂的查询            |

---

## 初步结果

| 智能体   | 准确率 | Token 数 | 成本  | 耗时 |
| -------- | ------ | -------- | ----- | ---- |
| SQL      | 100%   | 155K     | $0.51 | 45s  |
| Bash     | 52.7%  | 1.06M    | $3.34 | 401s |
| 文件系统 | 63%    | 1.28M    | $3.89 | 126s |

> SQL 明显占优：100%准确率，Bash 只有 52.7%

---

## 问题诊断

| 问题             | 说明                          |
| ---------------- | ----------------------------- |
| **性能瓶颈**     | 对 68,000 文件 stat()调用超时 |
| **缺少 schema**  | Bash 智能体不知道 JSON 结构   |
| **评估标准问题** | 标准答案本身有误              |

---

## 混合方案

| 策略       | 效果                   |
| ---------- | ---------------------- |
| SQL + Bash | 100% 准确率            |
| 先查后验   | 智能体自然发展验证策略 |

> 智能体先运行 SQL 查询，再用 grep 交叉验证

---

## 关键学习

| 学习                      | 说明               |
| ------------------------- | ------------------ |
| SQL 仍是结构化数据最优解  | 快、清晰、省 token |
| Bash 在探索验证环节有价值 | 检查文件、抽样核对 |
| 评估本身需要迭代          | 逐条追踪、修正基准 |
| 自我验证是关键            | 不是一次查询就结束 |

---

## 对智能体设计的意义

| 场景                    | 推荐                 |
| ----------------------- | -------------------- |
| Schema 清晰的结构化数据 | SQL 最直接高效       |
| 探索与验证环节          | Bash 很有价值        |
| 稳定结果                | 把自我验证纳入工作流 |

---

## 关键结论

> **真正的赢家是能否把自我验证纳入工作流，持续得到稳定结果**
