# 为 AI Coding Agents 编写高质量规范

> **作者**：Addy Osmani  
> **来源**：https://addyosmani.com/blog/good-spec/  
> **核心观点**：规范是 AI 执行的锚点，必须清晰、模块化、带自检机制

---

## 规范写作五大原则

| 原则               | 做法                                                  |
| ------------------ | ----------------------------------------------------- |
| **从高层愿景开始** | 先写简洁目标，让 AI 扩展为详细计划                    |
| **像 PRD一样构建** | 涵盖六个核心领域（命令、测试、结构、风格、Git、边界） |
| **模块化拆解**     | 一次给 AI 一个集中任务                                |
| **建立自检机制**   | 使用三层边界：始终做/先询问/永远不要                  |
| **迭代完善**       | 规范是活文档，持续更新                                |

---

## 六个核心领域检查清单

| 领域           | 示例                                |
| -------------- | ----------------------------------- |
| **命令**       | `npm test`、`pytest -v`（完整命令） |
| **测试**       | 框架、位置、覆盖率期望              |
| **项目结构**   | `src/` 应用、`tests/` 测试          |
| **代码风格**   | 一个代码片段胜过三段文字            |
| **Git 工作流** | 分支命名、提交格式、PR 要求         |
| **边界**       | 🚫 永远不要提交秘密                 |

---

## 三层边界系统

| 层级            | 含义           | 示例                   |
| --------------- | -------------- | ---------------------- |
| ✅ **始终做**   | 不询问即可执行 | 提交前运行测试         |
| ⚠️ **先询问**   | 需人工批准     | 修改数据库架构前先询问 |
| 🚫 **永远不要** | 硬停止         | 永远不要提交 API 密钥  |

---

## 规范驱动开发四阶段

| 阶段     | 动作                                    |
| -------- | --------------------------------------- |
| **指定** | 描述"做什么"和"为什么"，AI 扩展细节     |
| **规划** | 提供技术栈、架构、约束，AI 生成技术计划 |
| **任务** | AI 拆分为小、可审查的块                 |
| **实施** | 逐个处理，每阶段验证                    |

---

## 常见陷阱

| 陷阱                        | 后果           | 解决方案                 |
| --------------------------- | -------------- | ------------------------ |
| 模糊提示                    | AI 无锚定点    | 对输入、输出、约束要具体 |
| 过长上下文                  | 模型失焦       | 使用分层摘要或 RAG       |
| 跳过人工审查                | "纸牌屋代码"   | 始终审查关键路径         |
| 混淆 vibe coding 与生产工程 | 探索代码上生产 | 区分两种模式             |

---

## 关键结论

> **一个好的规范不仅告诉 AI 要构建什么，还帮助它自我纠正并保持在安全边界内。**
