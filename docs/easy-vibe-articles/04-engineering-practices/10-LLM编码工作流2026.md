# 我在 2026 年的 LLM 编码工作流

> **作者**：Addy Osmani  
> **来源**：https://addyosmani.com/blog/ai-coding-workflow/  
> **核心观点**：LLM 是"过度自信且容易犯错"的结对编程伙伴，需要强监督

---

## 工作流核心步骤

| 步骤           | 做法                               |
| -------------- | ---------------------------------- |
| **先写规格**   | 和 AI 一起梳理需求，整理成 spec.md |
| **拆小迭代**   | 每轮一个函数、一个 bug、一个功能   |
| **提供上下文** | 展示相关代码、约束、陷阱           |
| **选择模型**   | 不怕中途切换，尝试多个模型交叉检查 |
| **验证审查**   | 运行测试、人工审查、第二个模型批评 |

---

## Coding Agents 分类

| 工具                                   | 类型       | 特点                                 |
| -------------------------------------- | ---------- | ------------------------------------ |
| **Claude Code、Codex CLI、Gemini CLI** | CLI智能体  | 终端对话，读文件、跑测试、多步骤修复 |
| **Jules、Copilot Agent**               | 异步智能体 | 云端 VM 后台处理，开 PR              |

---

## 七个具体策略

| 策略                | 做法                                    |
| ------------------- | --------------------------------------- |
| **AI 卫生**         | 红队 AI 建议，测试棘手输入              |
| **基础知识不用 AI** | 设立"无 AI 日"保持基本技能              |
| **先自己尝试**      | 问 AI 前先尝试 15-30分钟                |
| **增强代码审查**    | AI 代码像审查同事代码一样               |
| **主动学习**        | 让 AI 解释不熟悉代码，追问"为什么"      |
| **学习日志**        | 跟踪常向 AI 求助的事项，弥补知识差距    |
| **AI 配对编程**     | 你写主体，AI 提建议；或 AI 起草，你重构 |

---

## 版本控制作为安全网

| 做法                   | 价值                               |
| ---------------------- | ---------------------------------- |
| **早且频繁提交**       | 每个小任务后清晰 git commit        |
| **分支隔离实验**       | git worktrees 防止 AI 会话相互干扰 |
| **commit作为"保存点"** | 出错可回滚到最近稳定状态           |
| **利用 git diff**      | 帮 AI 理解代码变更                 |

---

## 通过规则引导 AI

| 技术                      | 说明                                |
| ------------------------- | ----------------------------------- |
| **CLAUDE.md / GEMINI.md** | 项目级流程规则和偏好                |
| **自定义指令**            | 全局配置编码风格（Copilot、Cursor） |
| **内联示例**              | 展示一个类似函数，让 AI 模仿        |
| **"无幻觉"条款**          | 要求 AI 不确定时请求澄清而非编造    |

---

## 关键心态

| 心态                           | 说明                          |
| ------------------------------ | ----------------------------- |
| **LLM 是助手，不是自主编码者** | 你是高级开发者，AI 加速你     |
| **测试是力量倍增器**           | AI 在良好测试套件下"飞速"完成 |
| **自动化保持 AI诚实**          | CI/CD、linter、审查机器人     |

---

## 关键结论

> **我完全接受 AI 进入开发工作流，但前提是它必须处在经过认真设计、由人主导的框架里。AI 编码助手是力量倍增器，但人类工程师仍然是节目的导演。**
