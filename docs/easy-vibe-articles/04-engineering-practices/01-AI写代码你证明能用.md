# AI 写代码更快，你的工作是证明它能用

> **作者**：Addy Osmani  
> **来源**：https://addyosmani.com/blog/code-review-ai/  
> **核心观点**：最值钱的是"多快证明它真的可用"

---

## 开发者如何使用 AI 进行审查

| 方法                  | 说明                             |
| --------------------- | -------------------------------- |
| **临时 LLM 检查**     | 提交前将差异粘贴到 Claude/Gemini |
| **IDE 集成**          | Cursor/Claude Code 内联建议      |
| **PR 机器人和扫描器** | GitHub Copilot + Snyk 安全检查   |
| **自动化测试循环**    | AI 生成并运行测试，>70% 覆盖率   |
| **多模型审查**        | Claude 生成 + 安全模型审计       |

---

## 独立开发 vs 团队

| 维度         | 独立开发者              | 团队                  |
| ------------ | ----------------------- | --------------------- |
| **审查重点** | 关键路径 + 测试体系兜底 | 人工签字 + 共享上下文 |
| **速度瓶颈** | 推理时间（等待 AI）     | 审查成为速率限制器    |
| **前提条件** | 强测试体系              | 渐进主义 + 可消化提交 |

---

## 安全：AI 的可预测弱点

| 问题                  | 数据               |
| --------------------- | ------------------ |
| AI 生成代码含安全缺陷 | 约 45%             |
| 逻辑错误频率          | 人工代码的 1.75 倍 |
| XSS 漏洞频率          | 高 2.74 倍         |

> **如果代码涉及身份验证、支付、密钥或不受信任输入，就必须增加人工威胁建模审查**

---

## PR 契约

| 项目               | 说明                         |
| ------------------ | ---------------------------- |
| **什么/为什么**    | 1-2 句话说明意图             |
| **它能工作的证明** | 测试通过、截图/日志          |
| **风险 + AI 角色** | 层级以及哪些部分是 AI 生成的 |
| **审查重点**       | 1-2 个需要人工输入的领域     |

---

## 核心原则

| 原则                             | 说明                                  |
| -------------------------------- | ------------------------------------- |
| **先证明，再声称**               | 没有新测试或可验证演示的 PR，不应提交 |
| **AI 用作第一轮审查者**          | 将 AI 审查视为建议，不是最终仲裁者    |
| **人工审查集中在 AI 遗漏的内容** | 安全漏洞、重复代码、可维护性          |
| **强制执行增量开发**             | 小块提交作为检查点                    |
| **保持高测试标准**               | 有强测试实践的人从 AI 获益最多        |

---

## 关键结论

> **瓶颈已经从"写出代码"转向"证明代码有效"**
