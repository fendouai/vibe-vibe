# 工厂模型：Coding Agents 如何改变软件工程

> **作者**：Addy Osmani  
> **来源**：https://addyosmani.com/blog/factory-model/  
> **核心观点**：从"写代码的人"转向"设计生产系统的人"

---

## 抽象层次的演进

| 代际       | 特点           | 工作方式             |
| ---------- | -------------- | -------------------- |
| **第一代** | 加速的自动补全 | 你驾驶，工具辅助     |
| **第二代** | 同步智能体     | 你在场，实时指导     |
| **第三代** | 自主智能体     | 你定义结果，审查成果 |

> 第三代智能体可运行 30 分钟到几小时甚至几天

---

## 工厂心智模型

> **你不再只是编写代码。你在构建生产软件的工厂。**

| 组成要素       | 说明                                           |
| -------------- | ---------------------------------------------- |
| **智能体舰队** | 每个智能体有任务、工具箱、上下文、反馈循环     |
| **并行运作**   | 后端重构、功能实现、测试编写、文档更新同时进行 |
| **审查反馈**   | 审查输出、给出反馈、完善规范、重新部署         |

---

## 入职培训的平行

| 智能体行为   | 对应人类行为           |
| ------------ | ---------------------- |
| 接收规范     | 新工程师入职培训       |
| 分解子任务   | 探索代码库             |
| 搜索提交历史 | git blame 找负责人     |
| 向人类升级   | Slack/邮件请求领域知识 |

---

## 规范就是杠杆

| 问题         | 后果                     |
| ------------ | ------------------------ |
| 模糊思考     | 通过数十个并行智能体传播 |
| 糟糕架构决策 | 在整个舰队中扩散         |

> **你的规范质量决定输出质量。模糊的需求会产生模糊的实现。**

---

## 什么没有改变

| 传统技能 | 智能体时代                   |
| -------- | ---------------------------- |
| 清晰需求 | 必须以可评估方式阐明成功标准 |
| 强大抽象 | 干净架构变得更有价值         |
| 可靠测试 | 变成接近强制性               |
| 谨慎权衡 | 智能体不会自然平衡竞争关注点 |
| 人工监督 | 审查技能的标准提高了         |

---

## 测试的重要性

| 场景                   | 风险         |
| ---------------------- | ------------ |
| **单开发者跳过测试**   | 可管理       |
| **智能体舰队跳过测试** | 成本严重复合 |

> 红/绿 TDD：先写测试 → 确认失败 → 实现直到通过

---

## 未解决的问题

| 问题           | 说明                             |
| -------------- | -------------------------------- |
| **验证瓶颈**   | 智能体能生成，但验证是否正确更难 |
| **UI 验证**    | 视觉和行为回归仍脆弱             |
| **上下文窗口** | 大型代码库可能错过重要约束       |
| **不稳定环境** | 40 个智能体同时遇到同样问题      |

---

## 高杠杆工程师的新能力

| 能力           | 说明                           |
| -------------- | ------------------------------ |
| **系统思维**   | 在脑海中保持复杂架构           |
| **问题分解**   | 将模糊目标拆成范围明确的子任务 |
| **架构判断**   | 理解为什么系统这样设计         |
| **规范清晰度** | 编写明确、完整的需求           |
| **输出评估**   | 识别看起来正确但不正确的实现   |
| **编排技能**   | 管理多个并行工作流             |

---

## 关键结论

> **编程主要作为按键活动的时代已经结束。编程主要作为思考和判断活动的时代已经加速了几十年。**
