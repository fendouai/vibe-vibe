# Coding Agents 入门指南

> **作者**：Cognition Team (Devin)  
> **来源**：https://devin.ai/agents101  
> **核心观点**：任务委托、检查点与反馈机制整套方法

---

## 提示基础

### 核心原则

| 原则                 | 说明                             |
| -------------------- | -------------------------------- |
| **说清楚如何做**     | 不只说做什么，要说明偏好做法     |
| **告诉从哪里开始**   | 指明仓库、文档、关键组件         |
| **防御性提示**       | 提前补齐可能的歧义               |
| **给予访问工具权限** | CI、测试、类型检查器、代码检查器 |

### 示例对照

| 错误方式       | 正确方式                                 |
| -------------- | ---------------------------------------- |
| "添加单元测试" | 指定功能、边缘情况、需要模拟什么         |
| -              | "查看最新文档，在模型组目录创建实现文件" |

## 工作流程集成

### 立即接受新任务

| 场景                    | 操作                     |
| ----------------------- | ------------------------ |
| 队友问"能快速构建 X 吗" | 向代理发送提示调查或更改 |
| 副项目想法              | 委托代理，稍后回来处理   |

### 随时编码

| 场景             | 工具                         |
| ---------------- | ---------------------------- |
| 通勤时紧急错误   | Slack 移动应用或专门移动应用 |
| 意识到代码留错误 | wifi 不稳定也能解决          |

### 交出杂务

| 任务类型       | 说明                     |
| -------------- | ------------------------ |
| 二分查找旧提交 | 代理处理                 |
| 更新文档       | 代理处理                 |
| 重复性任务     | 节省时间，专注创造性工作 |

### 跳过分析瘫痪

| 场景             | 方法                         |
| ---------------- | ---------------------------- |
| 重构是否简化代码 | 让代理实现两个选项，比较决策 |
| 架构方法选择     | 具体例子比较，决策简单明了   |

## 委托大任务

### 自动化初稿

| 领域   | 起草者          | 完善者       |
| ------ | --------------- | ------------ |
| 新闻业 | 记者写初稿      | 编辑审阅润色 |
| 餐厅   | 配菜厨师准备    | 副厨师调味   |
| 编码   | 代理创建初稿 PR | 人审查完善   |

> **目标：省下 80% 时间，不追求 100% 自动化**

### 设置检查点

```
计划 → 实现块 → 测试 → 修复 → 检查点审查 → 下一个块
```

| 要求       | 说明                           |
| ---------- | ------------------------------ |
| 每阶段暂停 | 多层任务（数据库、后端、前端） |
| 澄清疑问   | "解释认证过程并确认安全性"     |
| 及早纠正   | 避免级联问题                   |

### 教它验证自己的工作

| 反馈方式     | 说明             |
| ------------ | ---------------- |
| 不只指出问题 | 清晰表达测试过程 |
| 重复测试模式 | 集成到永久知识库 |

## 自动化工作流程

### 创建快捷方式

| 常见任务       | 说明               |
| -------------- | ------------------ |
| 功能标志移除   | Playbooks 重复运行 |
| 依赖升级       | 强大可重用提示模板 |
| 新功能 PR 测试 | 自动触发多个代理   |

### 智能代码审查

| 工具                 | 说明                         |
| -------------------- | ---------------------------- |
| Greptile、CodeRabbit | 专业代码审查工具             |
| 自主代理             | 已索引仓库功能，提供准确洞察 |

### 钩入事件和警报

| 方式     | 说明                 |
| -------- | -------------------- |
| API 触发 | 响应特定事件自动触发 |
| MCPs     | 摄取第三方错误日志   |

⚠️ **生产服务问题**：让 AI 标记可疑错误，不要端到端修复

## 定制和性能

| 配置项       | 说明                               |
| ------------ | ---------------------------------- |
| **环境设置** | 语言版本、包依赖、自动检查对齐     |
| **CLI 工具** | 简单脚本提取工单信息、重启开发环境 |
| **知识库**   | 项目架构、测试类型、推荐工具       |

---

## 局限性

| 局限             | 说明                   |
| ---------------- | ---------------------- |
| **调试技能有限** | 错误定位需要更多上下文 |
| **视觉推理差**   | UI 设计匹配不够精细    |
| **知识截止**     | 新库需指向最新文档     |

---

## 管理时间和损失

| 策略           | 说明                               |
| -------------- | ---------------------------------- |
| **愿意止损**   | "忽略指令"或"兜圈子"时停止对话     |
| **多样化实验** | 尝试不同提示，加倍投入表现好的任务 |
| **必要时重开** | 新会话比混乱上下文更高效           |

---

## 安全和权限

| 实践             | 说明                        |
| ---------------- | --------------------------- |
| **创建账户**     | 一次性邮箱、自定义 IAM 角色 |
| **开发环境**     | 与团队工程师相同测试设置    |
| **只读 API Key** | 尽可能只读访问              |

## 关键结论

> **工程师被期望监督多个系统，同时保持深入理解和深思熟虑的判断**
