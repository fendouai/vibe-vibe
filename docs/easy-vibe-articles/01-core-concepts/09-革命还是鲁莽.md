# Vibe Coding：革命还是鲁莽放纵？

> **作者**：Addy Osmani  
> **来源**：https://addyo.substack.com/p/vibe-coding-revolution-or-reckless  
> **核心观点**：速度上瘾后更容易放弃对质量的要求

---

## Vibe Coding 的定义

| 定义来源      | 说明                                 |
| ------------- | ------------------------------------ |
| Karpathy 原意 | 靠直觉和交互式反馈推进的极端方式     |
| 误解          | 任何用了 AI 的编程都叫 vibe coding   |
| 正确理解      | **不阅读 AI 的代码就接受的特定风格** |

---

## 吸引力

| 吸引力         | 说明                             |
| -------------- | -------------------------------- |
| **速度**       | LLM 比熟练人类快一个数量级       |
| **创造性流动** | 与 AI 来回迭代的状态             |
| **门槛降低**   | 有想法但缺乏专业知识的人也能构建 |

---

## 安全和可维护性问题

| 问题类型     | 具体表现                    |
| ------------ | --------------------------- |
| **安全漏洞** | SQL 注入、XSS、认证授权缺失 |
| **可维护性** | 不理解自己的代码库          |
| **隐藏 Bug** | 打地鼠方法，处理眼前问题    |
| **技术债务** | 闪电般产生，抹去速度优势    |

> "疯狂地按'全部接受'按钮相当于用代码库玩俄罗斯轮盘赌"

---

## 原型 vs 生产

| 场景             | 适用性    |
| ---------------- | --------- |
| 个人项目、原型   | ✅ 适用   |
| 黑客马拉松演示   | ✅ 适用   |
| 学习和探索       | ✅ 适用   |
| 安全敏感生产系统 | ❌ 不适用 |
| 多协作者团队项目 | ❌ 不适用 |

> **最后 20%（通常是 80% 的努力）仍然需要传统工程**

---

## 对开发者的影响

| 变化         | 说明                               |
| ------------ | ---------------------------------- |
| **角色迁移** | 更多转向"指定要构建什么"           |
| **技能担忧** | 基础能力可能萎缩                   |
| **文化冲突** | 细致的工匠 vs 快速松散的 AI 驯服者 |
| **职业策略** | 学会负责任地使用 AI 工具           |

---

## 关键要点

| 要点             | 说明                      |
| ---------------- | ------------------------- |
| 不是编程终结     | 有特定优势和劣势的工具    |
| 不是无意义流行语 | 描述真实工作方式          |
| 环境很重要       | 黑客马拉松 ✅ 生产系统 ❌ |
| 需要驾驭         | 建立模式使其更安全有效    |

---

## 两全其美的方法

| 方法                 | 说明                         |
| -------------------- | ---------------------------- |
| **安全 vibe coding** | 沙盒执行、生成测试、明确界限 |
| **透明交互**         | Git diff 暂存，不盲目应用    |
| **保持理解**         | 使用 AI 更快编码，不外包理解 |

---

## 座右铭

> **"使用 AI 更快地编码，但永远不要外包你的理解。"**
