# Vibe Coding 不是低质量工作的借口

> **作者**：Addy Osmani  
> **来源**：https://addyo.substack.com/p/vibe-coding-is-not-an-excuse-for  
> **核心观点**：AI 不是放弃验证和维护责任的理由

---

## 残酷真相

> "两个工程师现在也许能制造出五十个人规模的技术债务"

| 问题             | 表现          |
| ---------------- | ------------- |
| **错误处理不足** | 缺少边界条件  |
| **性能边界不清** | 规模上崩溃    |
| **安全实践松散** | SQL 注入、XSS |
| **逻辑结构脆弱** | 纸牌屋代码    |

---

## 人类留在回路里

| 动作           | 说明                   |
| -------------- | ---------------------- |
| **阅读理解**   | 确认能解释结构与意图   |
| **重构**       | 分解为干净模块化部分   |
| **补边缘处理** | 空值检查、输入验证     |
| **加强类型**   | 将隐式假设转为显式契约 |
| **质疑架构**   | 检查是否低效方法       |
| **编写测试**   | 像审查任何 PR一样对待  |

---

## 关键规则

| 规则       | 说明                             |
| ---------- | -------------------------------- |
| **规则 1** | 始终审查 AI 生成的代码——没有例外 |
| **规则 2** | 建立编码标准并遵循               |
| **规则 3** | 使用 AI 加速，不是自动驾驶       |
| **规则 4** | 测试、测试、测试                 |
| **规则 5** | 迭代和完善，不接受不满意输出     |
| **规则 6** | 知道何时说不                     |
| **规则 7** | 记录和分享知识                   |

---

## 适用场景对照

### ✅ 很好的用例

| 场景         | 说明                       |
| ------------ | -------------------------- |
| 快速原型设计 | 验证想法，代码粗糙也没关系 |
| 一次性脚本   | 低风险，崩溃不是世界末日   |
| 学习和探索   | 新语言/API 示例生成        |
| 样板繁重任务 | 机械重复，AI 擅长          |

### ❌ 不太好的用例

| 场景           | 说明                       |
| -------------- | -------------------------- |
| 企业级软件     | 业务逻辑、并发、安全、合规 |
| 长期可维护项目 | AI 生成的大杂烩是糟糕基础  |
| 关键算法优化   | 人类独创性和深入理解仍优越 |
| 可解释性优先   | 审计员可阅读推理的代码     |

---

## 负责任做法

| 做法            | 说明               |
| --------------- | ------------------ |
| AI = 高速协作者 | 不是责任承接者     |
| 理解验证认可    | 才能进入代码库     |
| 人掌控设计      | AI 实现苦力活      |
| 记录意图        | 未来维护者不应猜测 |

---

## 底线

> **没有质量的速度毫无意义。更快地发布有错误、不可维护的代码是虚假胜利。**
