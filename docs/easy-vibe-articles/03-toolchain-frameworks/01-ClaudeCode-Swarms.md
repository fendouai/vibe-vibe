# Claude Code Swarms：多智能体协作开发

> **作者**：Addy Osmani  
> **来源**：https://addyosmani.com/blog/claude-code-agent-teams/  
> **核心观点**：角色分工与上下文隔离解决复杂任务

---

## 为什么多智能体协调很重要

| 问题                 | 解决                 |
| -------------------- | -------------------- |
| 单智能体上下文退化   | 第2步细节模糊到第5步 |
| LLM 随上下文扩展变差 | 信息越多越难聚焦     |

> **LLM 的表现随着上下文扩展而变差**

---

## 智能体团队架构

| 组件           | 角色                                |
| -------------- | ----------------------------------- |
| **团队负责人** | 创建团队、生成成员、协调工作        |
| **团队成员**   | 独立 Claude Code 实例，处理分配任务 |
| **任务列表**   | 带依赖跟踪和自动解锁                |
| **邮箱**       | 智能体间直接消息                    |

---

## 子智能体 vs 智能体团队

| 维度           | 子智能体         | 智能体团队                 |
| -------------- | ---------------- | -------------------------- |
| **上下文**     | 结果返回给调用者 | 完全独立                   |
| **通信**       | 仅向主智能体报告 | 团队成员直接相互消息       |
| **协调**       | 主智能体管理一切 | 共享任务列表 + 自我协调    |
| **最适合**     | 专注任务         | 需要讨论和协作的复杂工作   |
| **Token 成本** | 较低             | 较高（每个成员是独立实例） |

---

## 表现出色的场景

| 场景               | 说明                                   |
| ------------------ | -------------------------------------- |
| **调试的竞争假设** | 五个团队成员调查不同理论，辩论收敛     |
| **并行代码审查**   | 安全、性能、测试覆盖率各由不同成员负责 |
| **跨层功能工作**   | 前端、后端、测试各由不同成员负责       |
| **研究和探索**     | 多团队成员同时调查不同方法             |

---

## 控制团队

| 功能           | 说明                                 |
| -------------- | ------------------------------------ |
| **进程内模式** | 所有团队成员在主终端运行             |
| **分屏模式**   | 每个成员通过 tmux 或 iTerm2 获得窗格 |
| **计划批准**   | 团队成员只读模式，负责人批准后实施   |
| **委派模式**   | Shift+Tab 限制负责人仅协调           |

---

## 管理相似之处

| 管理原则             | 智能体编排                         |
| -------------------- | ---------------------------------- |
| **任务大小很重要**   | 太小=协调开销，太大=长时间无检查点 |
| **文件所有权很重要** | 不同团队成员拥有不同文件集         |
| **上下文加载很重要** | 生成提示中包含特定于任务的详细信息 |

---

## 需要注意的事项

| 问题                       | 说明                       |
| -------------------------- | -------------------------- |
| 负责人有时会实施而不是委派 | 使用委派模式限制           |
| 进程内团队成员没有会话恢复 | `/resume` 不会恢复         |
| 任务状态可能滞后           | 检查工作是否实际完成       |
| Token 成本随团队成员扩展   | 常规任务用单会话更成本效益 |

---

## 关键结论

> **让问题指导工具，而不是相反。如果在专注会话中的单个智能体能更快让你到达那里，就使用它。如果你需要并行专家，使用智能体团队。**
