# 如何为 AI 智能体写出优秀的规范

> **作者**：Addy Osmani  
> **来源**：https://addyosmani.com/blog/good-spec/  
> **核心观点**：写规范的目标是给 AI 一个足够清晰、细节充分、但又不过载的执行依据

---

## 五个核心原则

| 原则               | 做法                                                | 避免的错误             |
| ------------------ | --------------------------------------------------- | ---------------------- |
| **从高层愿景开始** | 先写简洁的目标描述，让 AI 扩展为详细计划            | 一开始就塞满所有细节   |
| **像 PRD一样构建** | 使用六个核心领域：命令、测试、结构、风格、Git、边界 | 松散的笔记堆           |
| **模块化拆解**     | 一次给 AI 一个集中任务，而非一个巨型提示            | 将整个项目塞进单个提示 |
| **建立自检机制**   | 使用三层边界系统：始终做/先询问/永远不要            | 简单的禁止列表         |
| **迭代完善**       | 规范是活文档，持续测试、反馈、更新                  | 写完就忘               |

---

## 六个核心领域（检查清单）

| 领域           | 内容示例                                             |
| -------------- | ---------------------------------------------------- |
| **命令**       | `npm test`、`pytest -v`、`npm run build`（完整命令） |
| **测试**       | 框架、文件位置、覆盖率期望                           |
| **项目结构**   | `src/` 应用代码、`tests/` 测试、`docs/` 文档         |
| **代码风格**   | 一个真实代码片段胜过三段描述文字                     |
| **Git 工作流** | 分支命名、提交消息格式、PR 要求                      |
| **边界**       | 🚫 永远不要提交秘密或编辑 node_modules/              |

---

## 三层边界系统

| 层级            | 含义           | 示例                     |
| --------------- | -------------- | ------------------------ |
| ✅ **始终做**   | 不询问即可执行 | "提交前运行测试"         |
| ⚠️ **先询问**   | 需要人工批准   | "修改数据库架构前先询问" |
| 🚫 **永远不要** | 硬停止         | "永远不要提交 API 密钥"  |

---

## 规范驱动开发四阶段

| 阶段        | 动作                   | 输出     |
| ----------- | ---------------------- | -------- |
| **1. 指定** | 描述"做什么"和"为什么" | 详细规范 |
| **2. 规划** | 提供技术栈和约束       | 技术计划 |
| **3. 任务** | 拆分为小而可审查的块   | 任务列表 |
| **4. 实施** | 逐个处理任务           | PR 审查  |

---

## 常见陷阱

| 问题                        | 后果                             | 解决方案                 |
| --------------------------- | -------------------------------- | ------------------------ |
| 模糊的提示                  | "给我构建一些很酷的东西"无锚定点 | 对输入、输出、约束要具体 |
| 过长上下文                  | 模型失去焦点                     | 使用分层摘要或 RAG       |
| 跳过人工审查                | "纸牌屋代码"容易崩溃             | 始终审查关键代码路径     |
| 混淆 vibe coding 与生产工程 | 探索代码直接上生产是自找麻烦     | 区分两种模式             |

---

## 关键结论

> **好的规范不仅告诉 AI 要构建什么，还帮助它自我纠正并保持在安全边界内。**
