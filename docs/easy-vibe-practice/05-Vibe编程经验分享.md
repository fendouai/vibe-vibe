# Vibe 编程经验分享：从执行者到决策者

> **面向人群**：具备编程能力的开发者  
> **核心主题**：如何用 AI 提升效率，同时保持对系统的掌控

---

## 本章目标

- 理解 Vibe Coding 的两种角色定位
- 掌握"方案先行、小步慢走"方法论
- 学会人机协作的正确分工
- 识别并应对 AI 的常见问题

---

## 两种 Vibe Coding 定位

| 角色             | 特点                           | 优势     | 风险                         |
| ---------------- | ------------------------------ | -------- | ---------------------------- |
| **零基础新人**   | 自然语言描述想法 → AI 生成代码 | 快速产出 | 系统黑盒、bug 横飞、原地打转 |
| **有经验开发者** | AI 作为执行层，人把控决策      | 高效可控 | 需主动定位问题               |

**本文定位**：第二种 —— 让 AI 成为高效的"执行层"

---

## 核心方法论

### 方法一：方案先行，实现为后

**原则**：先让 AI 输出技术方案 → 人 review → 再写代码

| 传统方式             | 方案先行             |
| -------------------- | -------------------- |
| 一句话让 AI 写代码   | 先让 AI 输出实现方案 |
| AI 边想边写          | AI 有明确锚点        |
| 逻辑断层、前后不一致 | 更可控、bug 更少     |

**为什么有效**：

- AI 自己生成的方案，自己更容易理解
- 把思路写清楚 → 写代码时有锚点
- 本质是"思维链"应用

**落地步骤**：

| 步骤 | 操作                                       |
| ---- | ------------------------------------------ |
| 1    | 描述业务实体 → AI 出表结构 → 审查调整      |
| 2    | 一次性生成 DAO → Service → Controller 三层 |
| 3    | 生成接口文档 → 生成前端页面                |
| 4    | 跑一遍增删改查做端到端验证                 |
| 5    | 遇问题定位到层 → 告诉 AI 改                |

### 方法二：小步慢走，步步为营

**原则**：大功能拆小任务 → 每步验证 → 同步文档

| 做法     | 说明                         |
| -------- | ---------------------------- |
| 功能拆分 | 每个任务明确目标、方案、步骤 |
| 即时更新 | 完成后同步文档和注释         |
| 每步验证 | 单测、功能测试、demo 页面    |

---

## 分形文档结构

> 让上下文永远保持最新

```
根目录主 md 文件
    ↓ 任何更新必须同步子文档
每个文件夹：
    ↓ 极简架构说明（三行）
    ↓ 列出文件名、地位、功能
    ↓ "文件夹变化请更新我"
每个代码文件开头：
    ↓ input（依赖什么）
    ↓ output（提供什么）
    ↓ pos（系统地位）
    ↓ "我被更新请更新注释和所属 md"
```

**效果**：AI 修改任何文件 → 自动触发文档同步链

## 实战案例：流媒体文件管理系统

### 项目概况

| 维度         | 内容                   |
| ------------ | ---------------------- |
| **定位**     | 一站式媒体资源管理平台 |
| **核心功能** | 媒资管理 + 权限管理    |

### 功能模块

| 模块         | 功能                                   |
| ------------ | -------------------------------------- |
| **权限体系** | 三档角色隔离、API Key 管理、第三方集成 |
| **媒体管理** | 多格式、自动转码、元信息提取、多视图   |
| **体验细节** | 批量操作、播放记录、备份下载、搜索过滤 |

### 开发策略

| 策略         | 说明                                  |
| ------------ | ------------------------------------- |
| 自上而下设计 | 脑中构思架构 → 输出方案文档           |
| 自底向上实现 | 底层模型 → 基础模块 → 功能模块 → 集成 |

## 人机协作分工

| 环节         | 人                           | AI                             |
| ------------ | ---------------------------- | ------------------------------ |
| **方案设计** | 架构决策、权限边界、技术选型 | 方案评审、可行性分析           |
| **代码实现** | 方案审查、代码审查、关键排错 | 体力活（增删改查、文档、界面） |
| **问题定位** | 根因分析、问题诊断           | 基于指导修复                   |
| **测试验证** | 测试设计、边界场景补充       | 测试脚本、demo 页面            |

**关键原则**：

- 不必逐行读懂代码，但必须清楚模块职责与变更影响
- 自底向上比"PRD 一把梭"更可控
- AI 擅长体力活，人擅长判断与闭环

---

## 效率提升

| 手写占比     | 内容                                               |
| ------------ | -------------------------------------------------- |
| **约 1%**    | 文案微调、无用 import 清理、幻觉编译错误、逻辑微调 |
| **节省 99%** | 编码时间 → 精力集中在架构和决策                    |

## 常见问题与应对

| 问题               | 表现                          | 应对                             |
| ------------------ | ----------------------------- | -------------------------------- |
| **API 幻觉**       | 虚构不存在的方法/参数         | 手动核实，明确"用官方 API"       |
| **错误假设打补丁** | 基于错误假设反复修改越改越乱  | 定位根因，明确指出"假设错了"     |
| **过度设计**       | 过多设计模式、工厂函数        | 代码审查时删除                   |
| **代码卫生**       | 无用 import、冗余函数、死代码 | 手动清理，强调"简洁无冗余"       |
| **边界遗漏**       | 只实现 happy path             | 测试发现补充，或提前列举边界场景 |

---

## 最大坑：无效调试循环

### 问题表现

```
第一次修改：改了 A（错误假设）
第二次修改：改了 B（A 引发新问题）
第三次修改：改了 C（B 引发新问题）
...
第 N 次：代码越来越乱，根因未解决
```

### 为什么会这样

| 原因       | 说明                           |
| ---------- | ------------------------------ |
| 缺乏元认知 | AI 无法"思考自己的思考"        |
| 无持久记忆 | 每轮对话独立，无法保持完整认知 |
| 错误堆砌   | 第一步假设错了，后续全错       |

### 唯一解决办法：亲自下场

| 步骤 | 操作                                              |
| ---- | ------------------------------------------------- |
| 1    | 识别危险信号：同一问题修改 2-3 次未解决 → 停止    |
| 2    | 自己定位根因：日志、断点、代码追踪                |
| 3    | 明确告诉 AI 根因："之前的假设错了，真正问题是..." |
| 4    | 让 AI 基于正确理解重新生成                        |

---

## 方向盘必须在人手里

| 环节     | 人必须做的               |
| -------- | ------------------------ |
| 架构层面 | 把控方向、模块边界清晰   |
| 问题定位 | 亲自分析、定位、理解根因 |
| 代码审查 | 发现过度设计、错误假设   |
| 测试阶段 | 主动测试、补充边界场景   |

**核心结论**：

> 发挥 AI 优势（高速输出）+ 规避 AI 劣势（无元认知、无持久记忆）
