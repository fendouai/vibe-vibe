## AI 编程的经济学

### 核心原则

**Token 就是钱**

- 每次让 AI 读取文件、搜索代码、生成回答，都在消耗费用
- 上下文越大，花费越高
- 精准的上下文不仅省钱，还能让输出更准确

### 优化提示词

优化提示词不是"打磨话术"，而是**减少 AI 需要读取的上下文范围**。

| ❌ 低效写法                | ✅ 高效写法                                |
| -------------------------- | ------------------------------------------ |
| "项目有问题，帮我检查一下" | "检查 `src/auth.ts` 中的登录逻辑"          |
| "我想添加一个用户系统"     | "在 `app/users/` 目录下添加用户 CRUD 功能" |
| "让我看看完整的项目结构"   | "只看 `app/` 和 `lib/` 目录"               |

### 成本优化策略

- **指定文件路径**比模糊描述更好
- **指明功能范围**比笼统说"项目有问题"更聚焦
- **删除客套话**，不设定专家角色，直接说任务
- **按需读取**，不要让 AI 读取整个项目

## VibeCoding 工作流

### 标准开发流程（5 步）

```
探索项目结构 → 规划实现步骤 → 编码 → 测试验证 → 提交
```

1. **探索项目结构**：了解代码组织，避免重复工作
2. **规划实现步骤**：列出任务清单，明确优先级
3. **编码**：按步骤实现，每完成一步验证
4. **测试验证**：确保功能正常，不引入新问题
5. **提交**：存档代码，便于回退

### 为什么需要标准流程

- **避免返工**：不会写到一半发现文件已存在
- **提高效率**：明确步骤，不迷路
- **降低风险**：分步验证，问题早发现

### 提示词心法

#### 给 AI 一个"出口"

告诉 AI："如果不确定，就明确说出来，等待我的确认，而不是强行编造。"

#### 避免强行编造

```
✅ "如果不确定数据结构，先问我再写代码"
❌ "你自己根据业务逻辑设计"
```

## Git 版本控制

### 为什么需要 Git

AI 编程很激进，可能为了修一个 Bug 而破坏三个旧功能。

**必须配置 Git，建立高频的本地版本记录。**

### 提交策略

```
AI 写完功能 → 自动存档 → 代码崩了随时回退
```

### 标准提交信息

**"每当你完成一个独立功能的开发，或修复完一个 Bug 并验证通过后，请自动运行 git commit 提交代码，并生成一句简洁的中文 commit message。"**

### 项目规则配置

在项目根目录创建项目规则文件（如 `.iderules`）：

```
禁止使用 any 类型
强制使用 pnpm
组件文件使用 PascalCase
工具函数使用 camelCase
```

AI 每次写代码前会先参考这份规范，生成代码质量明显提升。

## MCP、插件与 Skills

### 三种扩展方式

| 方式       | 作用                 | 复杂度 | 推荐场景                  |
| ---------- | -------------------- | ------ | ------------------------- |
| **插件**   | 扩展容器（一键安装） | 低     | 代码智能（LSP）、设计工具 |
| **MCP**    | 连接外部服务         | 中     | 数据库、API、GitHub       |
| **Skills** | 自动化工作流         | 高     | 自定义业务流程            |

### 核心原则

**能用插件的就不手动配置 MCP，能用内置的就不扩展。**

### 推荐插件

| 插件名称        | 功能说明              |
| --------------- | --------------------- |
| TypeScript-lsp  | TypeScript 语言服务器 |
| frontend-design | 前端界面设计          |
| feature-dev     | 功能开发向导          |
| github          | GitHub 操作           |
| code-review     | 代码审查              |
| playwright      | 浏览器自动化          |

### MCP 配置

MCP (Model Context Protocol) 让 AI 能连接外部服务：

- 数据库：查询数据、执行 SQL
- API：调用外部接口
- GitHub：操作仓库、查看 PR
- 文件系统：读写文件

### Skills

定义专属指令，自动化工作流：

```
"每当我说'分析数据'时，自动运行分析脚本"
"每当我说'部署'时，执行 pnpm build 和部署命令"
```

## 高效调试心法

### 调试沟通公式

**完整日志 + 操作步骤 + 预期结果**

### 示例

#### ❌ 低效写法

```
"报错了，帮我看看"
```

AI 需要追问细节，来回 3 轮才进入正题。

#### ✅ 高效写法

```
我运行 pnpm dev 启动项目，终端报错：

[完整错误日志]

我预期的结果是：开发服务器正常启动，能在 localhost:3000 访问

帮我分析并修复这个问题
```

AI 直接定位问题，节省 2-3 轮沟通。

### 循环修复模式

第一轮没解决？正常。继续给新日志：

```
按你的方法改了，现在出现新的错误：

[新错误日志]

请继续分析
```

**通常 2-3 轮解决**，不要放弃。

### 终极大招：让 AI 自己 build

```
"帮我运行 pnpm build，如果报错就自己修，直到 build 成功"
```

## 推荐配置

### 1. 安装 Claude Code

```bash
npm install -g @anthropic-ai/claude-code
claude

### 2. 安装推荐插件

打开 Claude，输入 `/plugin` 打开插件商店：

| 插件名称        | 功能说明              |
| --------------- | --------------------- |

brew tap farion1231/ccswitch
brew install --cask cc-switch
```

选择国内模型（如 GLM），更便宜且访问快。

## 常见问题

### Q：为什么我的 AI 总是胡说八道？

没有给 AI 一个"出口"。告诉它："如果不确定，就明确说出来，而不是强行编造。"

### Q：如何让 AI 遵循我的编码规范？

创建项目规则文件（如 `.iderules`），写上编码规范。

### Q：MCP 和插件有什么区别？

| MCP          | 插件                 |
| ------------ | -------------------- |
| 需要手动配置 | 一键安装             |
| 连接外部服务 | 功能包（可包含 MCP） |
| 更灵活       | 更简单               |

### Q：调试时应该给 AI 什么信息？

**完整的错误日志 + 操作步骤 + 预期结果**

不要只给最后一行错误信息。

### Q：AI 第一次没修好怎么办？

继续描述你尝试后的结果："我按你的方法改了，但现在出现了新的错误..."

通常 2-3 轮就能解决。
