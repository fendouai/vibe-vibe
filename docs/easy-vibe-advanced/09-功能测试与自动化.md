---

## 为什么需要测试

### "打地鼠"困境

改了评分功能 → 搜索坏了 → 修搜索 → 详情页坏了 → 修详情页 → 评分又坏了

三天，三个 bug，每个都是修前一个时引入的

### 测试的作用

**安全网**——每次改完代码，自动检查所有功能是否正常

---

## 测试金字塔

| 测试类型     | 速度 | 成本 | 数量 |
| ------------ | ---- | ---- | ---- |
| **单元测试** | 快   | 低   | 多   |
| **API 测试** | 中   | 中   | 中   |
| **E2E 测试** | 慢   | 高   | 少   |

**原则**：底层测试多，上层测试少

---

## API 测试

### 测试内容

- 接口返回是否正确
- 参数校验是否生效
- 错误处理是否合理

### 工具

使用 Postman 或编写自动化测试脚本

---

## E2E 测试

### 什么是 E2E 测试

**端到端测试**——模拟真实用户操作流程

### 工具

**Playwright** —— 自动化浏览器测试

### 测试内容

- 用户能否正常登录
- 搜索功能是否正常
- 提交表单是否成功

---

## Flaky 测试

### 什么是 Flaky 测试

**不稳定测试**——有时通过，有时失败，不是代码问题

### 原因

- 网络延迟
- 异步操作未完成
- 外部服务不稳定

### 解决方法

- 加等待时间
- 重试机制
- 排查异步问题

---

## 自动化工作流

### Git Hooks

提交代码时自动运行测试

### GitHub Actions CI

推送代码时自动构建和测试

### TDD（测试驱动开发）

先写测试 → 写代码 → 测试通过
