持久化 = 让数据在关闭或刷新后依然存在

## 关系型数据库基础

### 核心概念

| 概念                    | 含义               |
| ----------------------- | ------------------ |
| **Table（表）**         | 一个 Excel Sheet   |
| **Row（行）**           | 一条具体数据       |
| **Column（列）**        | 表头——定义数据属性 |
| **Primary Key（主键）** | 每行的唯一身份证号 |
| **Foreign Key（外键）** | 关联其他表的线索   |

---

## Drizzle Schema

### 示例代码

```typescript
import { pgTable, serial, text, timestamp, integer } from 'drizzle-orm/pg-core'

export const users = pgTable('users', {
  id: serial('id').primaryKey(), // 自增主键
  email: text('email').notNull().unique(), // 必填且唯一
  name: text('name'), // 可选字段
  createdAt: timestamp('created_at').defaultNow()
})

export const posts = pgTable('posts', {
  id: serial('id').primaryKey(),
  title: text('title').notNull(),
  authorId: integer('author_id').references(() => users.id) // 外键关联
})
```

### 常用类型

| 类型          | 用途                                |
| ------------- | ----------------------------------- |
| `serial`      | 自增整数                            |
| `text`        | 文本（永远用 text，不要用 varchar） |
| `boolean`     | 布尔                                |
| `timestamptz` | 带时区时间                          |
| `integer`     | 整数                                |
| `numeric`     | 精确小数（金额必须用这个）          |

## 数据库操作

### CRUD

| 操作       | 含义 |
| ---------- | ---- |
| **Create** | 增   |
| **Read**   | 查   |
| **Update** | 改   |
| **Delete** | 删   |

### 事务

**多个操作打包成"要么全成功、要么全失败"**

用于：转账、订单等关键业务

### 索引

**像书本的目录——加速查询**

- 没有索引：全表扫描
- 有索引：直接定位，速度快几十倍
- **只在经常查询的字段上建索引**

## 数据校验三道防线

| 防线       | 位置          | 作用                                       |
| ---------- | ------------- | ------------------------------------------ |
| **第一道** | 数据库约束    | `.notNull()`、`.unique()`、`.references()` |
| **第二道** | 后端 API 校验 | 处理请求时检查格式、范围                   |
| **第三道** | 前端表单验证  | 用户提交前快速反馈                         |

**别因为有了前端校验就省略后端和数据库层面**

## PostgreSQL 选择

### 为什么选 PostgreSQL

- OpenAI ChatGPT 后端用的就是 PostgreSQL
- 支持亿级用户，每秒百万查询
- **JSONB 支持**：像 NoSQL 一样存 JSON
- **pgvector**：向量检索——AI 长期记忆（RAG）核心技术

## 实战避坑

### 开发 vs 生产数据库

|              | 建议                   |
| ------------ | ---------------------- |
| 学习阶段     | 可用一套数据库快速上手 |
| 有真实用户后 | 分离开发和生产环境     |

### Connection URL 格式

```
postgresql://用户名:密码@主机地址:端口/数据库名
```

**任何标点符号错了都会连接失败**

## AI 交叉论证法

让 AI 设计表结构后，发给另一个 AI 检查：

```
作为资深数据库架构师，
这个设计是否合理？
有什么潜在的性能隐患或逻辑漏洞？
```

通常两轮"左右互搏"就能得到健壮的数据库模型
