## 产品验证三步法

### 第一步：灵魂三问

| 问题       | 常见错误       | 正确做法       |
| ---------- | -------------- | -------------- |
| 用户是谁？ | "所有人"       | 明确具体人群   |
| 痛点在哪？ | "我觉得需要"   | 基于真实场景   |
| 为何用你？ | "有最好的技术" | 用户关心的价值 |

### 第二步：MVP 思维

**能验证假设的最简版本是什么？**

- 不要想着一次做出完美产品
- 先做最小可行产品（MVP）
- 快速验证核心假设

### 第三步：快速验证

- 用最小成本验证假设
- 可以先用手工方式服务
- 或做一个简单落地页收集反馈
- **失败得越早，成本越低**

## 与 AI 确认需求

### 核心原则

**不要等 AI 来问你，而是让 AI 明确说出它的理解**

### 确认模板

每次讨论完需求后，让 AI 复述：

```
请确认以下理解是否正确：
1. 目标用户是谁？
2. 核心功能有哪些？
3. 不做哪些功能？
4. 可能有什么问题？
```

### 为什么需要确认

- AI 不会主动提问
- AI 基于训练数据做假设
- 如果假设错了，后续迭代成本更高

## PRD 编写实战

### PRD 五个核心部分

| 部分           | 内容                                           |
| -------------- | ---------------------------------------------- |
| 文档信息       | 版本、阶段、干系人、迭代历史                   |
| 需求背景与目标 | 项目概述、核心问题、用户故事、项目目标         |
| 方案概述       | 核心业务流程、功能流程、信息架构（Mermaid 图） |
| 细节方案       | 页面原型、交互说明、边缘 Case、非功能性需求    |
| 上线计划       | 排期、灰度发布策略                             |

### PRD 迭代过程

| 版本 | 重点                                              |
| ---- | ------------------------------------------------- |
| 初稿 | "为什么做"——需求背景、目标、核心价值              |
| 中稿 | "是什么"——核心业务流程、功能流程图、原型交互      |
| 定稿 | "怎么做"——最终 UI 设计、边缘 Case、埋点、上线计划 |

## 需求管理技巧

### 需求范围管理

明确 "In-Scope（范围内）" 和 "Out-of-Scope（范围外）"

### 需求优先级排序

| 需求 ID | 模块 | 描述     | 优先级 | 状态   |
| ------- | ---- | -------- | ------ | ------ |
| R001    | 用户 | 注册登录 | P0     | 待开发 |
| R002    | 内容 | 发布文章 | P1     | 待开发 |

### 用户故事格式

```
作为一名<角色>，
我想要<完成某项任务>，
以便于<实现某个价值>
```

### SMART 目标

- Specific（具体的）
- Measurable（可衡量的）
- Achievable（可实现的）
- Relevant（相关的）
- Time-bound（有时限的）

## 从 PRD 到代码

### AI 如何"阅读" PRD

- AI 不会"脑补"模糊的地方
- AI 严格按字面意思理解
- PRD 有歧义 → AI 要么猜测（可能错），要么停下来问

### PRD→代码流程

```
提取关键信息 → 构建数据模型 → 设计 API 接口 → 生成前后端代码
```

### 关键原则

**想清楚再动手，用文档训练思维**
