## 为什么需要反馈

### 你只是千千万万个用户中的一个

你的使用习惯、需求、痛点，可能和其他人完全不一样

### 案例

有个团队做了"自动同步云端"功能，用户最想要的其实是"离线查看"

如果早点收集反馈，就不会浪费几个月

---

## 收集反馈的渠道

| 渠道         | 适合收集什么 |
| ------------ | ------------ |
| 页面反馈按钮 | Bug 报告     |
| 邮箱         | 深度建议     |
| 社交群组     | 想法和讨论   |

### 引导用户

用户不知道该说什么，你需要引导：

- 遇到了什么问题？
- 想要什么功能？
- 有什么建议？

---

## 反馈分类

### 两个维度

| 维度         | 问题                 |
| ------------ | -------------------- |
| **影响范围** | 这个问题影响多少人？ |
| **严重程度** | 这个问题有多严重？   |

### 优先级

|            | 影响     | 严重度 | 优先级   |
| ---------- | -------- | ------ | -------- |
| 登录失败   | 一半用户 | 很严重 | **最高** |
| 颜色不好看 | 少数人   | 不严重 | **最低** |

---

## 用户访谈

### 为什么需要访谈

有些反馈很模糊，如"不好用"，无从下手

### 关键技巧

不要问"你喜欢吗"，要问"上次遇到什么困难"

让用户演示给你看，边用边说

### 数量

5-10 个就够了，会反复听到同样的问题

---

## 数据驱动决策

### 结合数据与反馈

数据告诉你发生了什么
反馈告诉你为什么发生

### 案例

数据：注册流失率高
反馈：表单太长

→ 简化注册流程

---

## 持续迭代

### 产品上线不是结束

上线第一天，才是真正开始验证想法

### 迭代思维

- 根据反馈调整方向
- 改进功能
- 放弃没人用的东西
- 加强用户喜欢的部分

**没有一次就能做到完美的产品**
