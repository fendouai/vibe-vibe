## 代码运行的三种状态

| 状态                   | 特点           | 用途       |
| ---------------------- | -------------- | ---------- |
| **Dev（开发）**        | 保存后自动更新 | 调试、开发 |
| **Build（构建）**      | 压缩优化代码   | 准备上线   |
| **Production（生产）** | 运行构建后版本 | 正式运行   |

## 技术栈选择框架

### 决策原则

**先搞清楚要做什么，再决定用什么做**

### 推荐方案

| 项目类型 | 推荐                                  |
| -------- | ------------------------------------- |
| 全栈应用 | Next.js（AI友好、生态成熟、部署便捷） |
| 纯前端   | Vite                                  |
| 数据库   | PostgreSQL（关系型+JSONB+pgvector）   |

### PostgreSQL 托管方式

| 平台         | 特点                          | 适用场景       |
| ------------ | ----------------------------- | -------------- |
| **Supabase** | 完整 BaaS（认证、存储、实时） | 快速开发 MVP   |
| **Neon**     | 专注数据库，无服务器架构      | 自定义后端需求 |

## 编程的基本构件

所有编程语言都基于四个核心概念：

| 构件     | 作用                 |
| -------- | -------------------- |
| **变量** | 存储数据             |
| **函数** | 封装可复用的逻辑     |
| **条件** | 根据不同情况采取行动 |
| **循环** | 批量处理重复任务     |

## PRD 与技术文档

| 文档类型                | 回答什么问题 | 内容                                   |
| ----------------------- | ------------ | -------------------------------------- |
| **PRD（产品需求文档）** | 做什么       | 用户是谁、核心功能、交互方式、边缘场景 |
| **技术文档**            | 怎么做       | 技术栈、数据库设计、API 定义、部署方案 |

## API 与 HTTP 基础

### HTTP 四个部分

| 部分         | 作用                            |
| ------------ | ------------------------------- |
| **请求方法** | GET/POST/PUT/DELETE——做什么操作 |
| **URL**      | 资源的地址                      |
| **Headers**  | 身份认证等元数据                |
| **Body**     | 发送的数据内容                  |

### 常见状态码

| 状态码 | 含义       | 原因             |
| ------ | ---------- | ---------------- |
| 401    | 未认证     | 需要登录         |
| 403    | 无权限     | 登录了但无权访问 |
| 404    | 资源不存在 | 地址错误或已删除 |

## 前后端分离

| 层         | 职责                                     |
| ---------- | ---------------------------------------- |
| **前端**   | 展示界面、收集用户输入、发送 HTTP 请求   |
| **API**    | 接收请求、验证参数、调用数据库、返回响应 |
| **数据库** | 存储和检索数据                           |

**前端不能直接访问数据库**——会暴露密码和数据

---

## 配置文件格式

| 格式     | 特点                     | 用途                 |
| -------- | ------------------------ | -------------------- |
| **JSON** | 大括号 `{}`、方括号 `[]` | 数据交换、API 响应   |
| **YAML** | 缩进表示层级             | CI/CD 配置、项目配置 |

## API 集成实战

### 六个步骤

1. 获取凭证（API Key）
2. 选择 SDK
3. 配置环境变量（`.env`）
4. 编写最小测试
5. 归档参考文档
6. 处理常见错误

### 关键提醒

**API Key 保存到环境变量 `.env` 文件中，不要写死在代码里**

## 模块化思维

不要把所有代码塞在一个文件里，拆分成不同模块：

| 模块         | 职责     |
| ------------ | -------- |
| `auth`       | 认证相关 |
| `api`        | 接口相关 |
| `components` | UI 组件  |
