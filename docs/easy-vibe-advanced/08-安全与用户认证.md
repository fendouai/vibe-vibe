## 密钥泄露案例

### 真实场景

`git push` 后 10 分钟收到邮件：

> "您的账户在过去 10 分钟内产生了 $127.43 的 API 调用费用"

### 原因

`.env` 文件被推到 GitHub，`OPENAI_API_KEY` 明晃晃地躺在那里

### 时间

从密钥出现在公开仓库到被盗用，平均只需要 **5 分钟**

### 关键提醒

- Git 记录了每一次提交的历史
- 删除文件只是删除最新版本
- 历史记录里密钥还在

## 密钥管理

### 环境变量

**API Key 保存到 `.env` 文件，不要写死在代码里**

### .gitignore

确保 `.env` 文件被 `.gitignore` 拦截，不会推到 GitHub

### 云端配置

部署后在平台的环境变量页面配置，平台会自动注入

## 用户认证

### 认证方式

| 方式         | 特点                         |
| ------------ | ---------------------------- |
| **Session**  | 传统方式，服务器存储登录状态 |
| **Token**    | 现代 API 常用，无状态        |
| **OAuth**    | 第三方登录（微信、GitHub）   |
| **Passkeys** | 无密码登录，未来趋势         |

### 推荐方案

**Better Auth** —— 类型安全的认证库，适合 Next.js

### 为什么不用自己写

- 认证逻辑复杂，容易出错
- 安全漏洞风险高
- 成熟方案已经解决了大部分问题

## 路由保护

### Middleware

Next.js 的 Middleware 可以拦截所有请求

### 保护方式

| 级别   | 方式                         |
| ------ | ---------------------------- |
| 页面级 | 检查用户是否登录，未登录跳转 |
| 接口级 | 检查请求是否带有效 Token     |

### CORS

跨域请求需要配置 CORS，否则会被浏览器拦截

## 进阶安全防护

### 常见攻击

| 攻击         | 防护                             |
| ------------ | -------------------------------- |
| **SQL 注入** | 使用 ORM，不手写 SQL             |
| **XSS**      | 不信任用户输入，使用框架自动转义 |
| **CSRF**     | 使用 Token 验证                  |

### AI 应用安全

- API Key 不要暴露给前端
- 用户输入要做限制
- 模型调用要有监控
