## 第一步：上线前检查（10 分钟）

### 为什么把上线放在最后

第1章做完时是**原型**，第6章是**正式亮相**。

上线应该建立在：

- ✅ 页面有基本样子
- ✅ 内容更完整
- ✅ 数字分身更像你

**只有到这个程度，链接才值得被别人点开**

### 上线前朴素自查

一个够用的上线前清单：

- [ ] 页面能正常打开，没有明显报错
- [ ] 手机端阅读没有严重错位
- [ ] 数字分身可以正常回复
- [ ] 你的名字、介绍、作品、联系方式已经不是空白
- [ ] API Key 没有写死在代码里，环境变量位置正确

**确保你不是把一个明显还没准备好的版本直接公开**

---

## 第二步：推代码到远程仓库（15 分钟）

### 最小路径

```
本地项目
  ↓
远程仓库（GitHub / Gitee）
  ↓
部署平台
  ↓
公网链接
```

### 为什么要先推远程仓库

远程仓库同时承担两件事：

- 它是你的代码备份
- 它也是部署平台读取代码的来源

### 最小操作步骤

1. 创建远程仓库
2. 把本地代码推上去
3. 在 EdgeOne Pages 连接仓库
4. 使用默认构建配置先跑一次
5. 拿到第一个可访问链接

**第一次部署只求跑通，不求理解全部配置**

### 如果部署失败

先看日志，再交给 AI：

```text
这是我在部署平台看到的构建日志。

请先帮我判断最可能的失败原因是什么。

只告诉我下一步先改哪一项，不要一次给太多方案。
```

## 第三步：上线后自测（10 分钟）

### 部署成功 ≠ 作品真的能用

### 自己走一遍真实环境

1. 链接能不能正常打开
2. 首屏和主要内容区有没有明显错位
3. 手机端读起来顺不顺
4. 数字分身能不能正常回复
5. 联系方式和作品信息是不是完整可见

**重点不在"严谨测试流程"，而在"不要把部署成功误以为作品已经完全没问题"**

## 第四步：收集第一轮真实反馈（15 分钟）

### 找 2-3 个人真实测试

只问三件事：

- 你打开后第一眼觉得我是做什么的？
- 你知不知道该点哪里、该问什么？
- 有没有哪里让你卡住，或者觉得不太自然？

### 反馈的意义

反馈不是否定，而是作品进入真实世界之后最自然的下一批素材。

**很多值钱的改进线索，只有别人真的点开链接之后才会出现**
